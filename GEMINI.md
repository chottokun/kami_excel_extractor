# Engineering Standards (Kami Excel Extractor)

本プロジェクトにおける開発およびメンテナンスの基準を定義します。

## 1. 非同期処理 (asyncio) の徹底
- I/O 待ちが発生する処理（LLM API 呼び出し、画像処理、ファイル操作の重いもの）は原則として非同期（`async` / `await`）で実装します。
- 同期コンテキストからの呼び出し用に、`asyncio.run` を内包したラッパーメソッドを公開しますが、ライブラリ内部の連鎖的な呼び出しでは直接非同期版を `await` し、二重 `asyncio.run`（RuntimeError）を回避します。

## 2. データの堅牢性と互換性
- **JSON シリアライズ**: 最終的な抽出結果（辞書/リスト）は、常に `json.dump` 可能な型のみで構成されることを保証します。`date` や `datetime` オブジェクトは、出力前に必ず ISO 形式の文字列へ変換します。
- **画像正規化**: エクセルから抽出された画像バイナリは、直接送信せず Pillow を介して標準的な PNG 形式に正規化します。これにより、VLM API (Gemini等) での読み取りエラー (400 BadRequest) を防止します。

## 3. レート制限と並列制御
- 外部 API へのリクエストは `GEMINI_RPM_LIMIT` 環境変数に基づいた `asyncio.Semaphore` によって並列度を制御します。
- レート制限を超過しないよう、かつリソースを最大限活用できるよう並列実行を設計します。

## 4. セキュリティ
- 一時ディレクトリやファイルは、`tempfile.TemporaryDirectory` や `tempfile.NamedTemporaryFile` を使用して安全に管理し、CWE-377 等の脆弱性を排除します。
- センシティブな情報（APIキー等）は環境変数から取得し、ログやコードにハードコードしないことを徹底します。

## 5. テスト基準
- LibreOffice などの OS 依存ライブラリが必要なため、テストは Docker コンテナ内での実行を正とします。
- 新機能の追加時には、必ず `pytest` による自動テストを追加し、特に非同期メソッドのモックが適切に行われていることを確認します。
- 実データ（`complex_report.xlsx` 等）を用いた統合テストを定期的に行い、VLM の出力揺れや形式変更に対する耐性を検証します。
