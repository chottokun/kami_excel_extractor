# 構造化データ抽出の信頼性向上計画（JSON移行とリトライロジック）

本計画は、ローカルLLMを用いた際の YAML パースエラーを削減し、システム全体の堅牢性を高めることを目的としています。

## 現象と課題
- `qwen3.5:4b` や `gemma4:latest` などの小型モデルにおいて、YAML の構文エラー（未引用のコロンや括弧の不一致等）が発生し、パースに失敗することがある。
- 現在はパースに失敗した場合、エラー内容をログに残して Markdown に記載するのみとなっており、データ抽出としては未完の状態で停止している。

## 改修内容

### 1. YAML から JSON への移行
ローカルLLMの「JSON Mode」を最大限活用できるよう、出力形式を JSON に変更します。
- プロンプトの要求形式を JSON に更新。
- `litellm.acompletion` において `response_format={"type": "json_object"}` を指定。
- `PyYAML` によるパースから、Python 標準の `json` モジュール（または Pydantic による自動パース）へ切り替え。

### 2. セルフリトライ機能の導入
パースエラーを検知した際、自動的にエラー内容をプロンプトに含めてLLMに再試行を依頼します。
- 初回試行：通常の抽出プロンプト。
- リトライ試行（1回）：エラーメッセージ（どの行でパースエラーが出たか等）を含めた「修正依頼」プロンプト。

### 3. Pydantic による構造定義とバリデーション
LLMからのレスポンスが規定の構造を維持しているか、実行時に検証します。
- `SheetData`, `RagChunk` 等の Pydantic モデルを定義。
- バリデーションエラー時もリトライロジックをトリガー。

## 期待される効果
- 構造化抽出の成功率が大幅に向上。
- JSON Mode による構文レベルのエラー（引用符の閉じ忘れ等）の撲滅。

## 今後の展望
- 精度が向上した時点で、実データを用いたさらなるバッチ検証。
- モデル（Provider）ごとに最適なプロンプトテンプレートの整備。
