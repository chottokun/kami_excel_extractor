# 画像プレビューのページネーション問題：解析と拡張設計

## 1. 課題の再確認
現在、抽出プロセス中の `pdftocairo` では `-singlefile` オプションを用いてLibreOffice (`soffice`) が出力した統合PDFの「先頭ページ（ページ番号1）」のみをPNG化している。
この仕様により、以下の課題となっている。
1. **他シートへの画像汚染**: Sheet 2やSheet 3の処理時にも、LLMには「Sheet 1の1ページ目（先頭ページ）」の画像が常に送られるため、視覚的コンテキストとデータ構造が不一致となる。
2. **長尺シートの切り捨て**: 1つのシートがPDF上で複数ページに跨る場合、2ページ目以降の視覚情報が無視され、全体のレイアウト情報が欠落する。

## 2. アプローチの比較と採用戦略

これを打開するために、各シート専用の完全な画像を生成する技術スタックを検証した。

| アプローチ | 手法概要 | 結果 |
| :--- | :--- | :--- |
| **A: フィルターオプション** | `soffice` の `--convert-to 'pdf:calc_pdf_Export:{"SinglePageSheets":...}'` | ❌ 動作せず（全201ページのPDFが生成されシート紐付け不可） |
| **B: PyUNO マクロ制御** | `soffice` をリスナーモードで起動し、Python連携でアクティブシートを切り替えてPDF出力 | ❌ コンテナ内通信の複雑化・安定性の懸念から却下 |
| **C: OpenPyXL 単一抽出** | 抽出対象「以外」のシートを `openpyxl` で削除して一時Excelファイルを作成し、それをPDF出力 | 🟢 **採用**（対象シート専用の正確なマルチページPDFが生成される） |

## 3. 実装のシステムアーキテクチャ

最も堅牢な **「C: OpenPyXL 単一抽出」** を用いて、パイプラインを拡張する。

### Step 1: Excel一時ファイルの分割
- 抽出ループの内部で、処理対象とするシート名のみを残した一時Excelファイル (`/tmp/isolated_{sheet_name}.xlsx`) を保存する。

### Step 2: 単独シートのPDF化と全ページPNG展開
- 一時ファイルを `soffice` を用いてPDF化する（このPDFはシートの長さに応じて1ページ〜十数ページ程度になる）。
- `pdftocairo` 実行時の `-singlefile` オプションを除去し、対象シートの全PDFページを連番のPNG画像（`{sheet_name}-1.png`, `{sheet_name}-2.png` 等）として出力させる。

### Step 3: LLMへのマルチモーダル・コンテキスト投入
- 出力された連番PNGをすべてBase64エンコードし、APIリクエスト時の `messages` ペイロードの該当シートの処理に「複数の画像配列」として同時投入する。
- 巨大な行数・列数を持つシートであっても、AIは複数の画像を跨いで全体構造を認識できる。

## 4. 拡張による期待効果
- このプロセスにより、外部ライブラリへの新たな依存を追加することなく、**画像と対象シートの完全な一致**が実現される。
- これによってVLMが「対象シートの完全な視覚的レイアウト」と「HTMLテーブルの座標的構造」を統合して推論できるようになり、解析精度の飛躍的な向上が見込める。
