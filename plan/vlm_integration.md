# **VLM統合とマルチモデル対応戦略**

LiteLLM ライブラリを中核に据え、OpenAI, Gemini, Azure OpenAI などの多様なVLMを共通のインターフェースで活用するための戦略を定義する。

## **1. マルチモデル・アブストラクション (LiteLLM)**

特定ベンダーのSDKに依存せず、LiteLLM を介して推論を実行する。

- **モデル名の標準化:** プロバイダープレフィックス（`openai/`, `gemini/`）を用いて動的にルーティング。
- **OpenAI互換インターフェース:** メッセージ構成や `response_format` をOpenAI標準に統一し、プロバイダー間の差異（APIの癖）をライブラリ層で吸収。

## **2. マルチモーダル・メッセージ構成**

VLMに対し、視覚（画像）と論理（座標マップ）を効果的に提示する。

- **Role: System:** 「Excel構造化のエキスパート」としてのペルソナを付与。
- **Role: User (Multi-modal):**
    - **Text Part:** 階層化された「座標マップ (JSON)」を提示。
    - **Image URL Part:** Base64エンコードされたPNG画像。
    - **Instruction Part:** 抽出ルールと、特定の写真や図解の解釈指示。

## **3. 精度向上のためのテクニック (実証済み)**

- **OCRの禁止:** 「画像内の文字を直接読み取らず、座標マップから値を引用せよ」と明示的に指示し、読み間違い（ハルシネーション）を構造的に排除。
- **文脈的マッピング:** 「座標 A10 の写真と、座標 B10 のキャプションを関連付けよ」といった空間的推論を強要。
- **JSON Modeの強制:** `response_format={"type": "json_object"}` を使用し、常にパース可能な構造化データを取得。

## **4. モデルの自動判別とフォールバック**

- **動的選択:** `genai.list_models()` (または各社API) を利用し、その時点で利用可能な最新かつ高速な Flash モデル（例：`gemini-2.5-flash`）を自動選択。
- **エラーハンドリング:** 404（モデル不在）や 429（レート制限）を検知し、安定した下位モデル（例：`gemini-1.5-flash`）へ自動的に切り替えるリトライロジックを実装。
