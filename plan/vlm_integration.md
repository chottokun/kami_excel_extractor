# **VLM統合とマルチモデル対応戦略**

LiteLLM ライブラリを中核に据え、OpenAI, Gemini, Azure OpenAI などの多様なVLMを共通のインターフェースで活用するための戦略を定義する。

## **1. マルチモデル・アブストラクション (LiteLLM)**

特定ベンダーのSDKに依存せず、LiteLLM を介して推論を実行する。

- **非同期呼び出し:** `litellm.acompletion` を使用し、I/O 待ちの間に他のシートの解析を進めることで、パイプライン全体の並列度を向上。
- **モデル名の標準化:** プロバイダープレフィックス（`openai/`, `gemini/`）を用いて動的にルーティング。
- **OpenAI互換インターフェース:** メッセージ構成や `response_format` をOpenAI標準に統一し、プロバイダー間の差異をライブラリ層で吸収。

## **2. マルチモーダル・メッセージ構成**

VLMに対し、視覚（画像）と論理（HTMLテーブル）を効果的に提示する。

- **Role: System:** 「Excel構造化のエキスパート」としてのペルソナを付与。
- **Role: User (Multi-modal):**
    - **Text Part:** 座標と結合セルの構造を保持した「HTMLテーブル」を提示。
    - **Image URL Part:** Base64エンコードされたPNG画像。
    - **YAML 指示:** `Markdown` のコードブロック (```yaml) を用いた YAML 形式での出力を指定。YAMLはJSONに比べトークン効率が良く、構造の解釈ミスが少ない。

## **3. 精度向上とコスト最適化のテクニック**

- **OCRの禁止:** 「画像内の文字を直接読み取らず、提供されたデータソース（HTML）から値を引用せよ」と明示的に指示。
- **ビジュアルサマリーのキャッシュ:** `aget_visual_summary` の結果をメモリ内にキャッシュ。同一の画像（Base64 URL）に対して重複した解析を防ぎ、APIコストを大幅に削減する。
- **JSON シリアライズの保証:** LLM から返された YAML や、内部のメタデータに含まれる `date`/`datetime` オブジェクトは、最終出力前に文字列へ変換される。

## **4. モデルの自動判別とフォールバック**

- **動的選択:** 環境変数 `GEMINI_MODEL` に基づき、利用可能な最新モデル（例：`gemini-2.0-flash`）を優先。
- **エラーハンドリング:** 404（モデル不在）や 429（レート制限）を検知し、適切なエラーログを出力。並列実行時はセマフォによりレート制限の回避を試みる。
